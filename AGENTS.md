# AGENTS.md

## What this repo is

Go CLI (`cmd/orchestra`) that orchestrates multiple AI coding agents (Claude Code, OpenCode, Mimo, etc.) through a single supervised interface. Agents run headless; human approves via diff review.

## Build & verify (run these in order before pushing)

```sh
go build ./...          # compile
go vet ./...            # static analysis
go test ./...           # all tests
gofmt -w .              # format (CI enforces this)
```

No shortcuts. CI runs exactly these four steps against Go 1.22.

## Single-focus commands

```sh
go test ./internal/scheduler/...   # test one package
go vet ./internal/engine/...       # vet one package
gofmt -l .                         # list unformatted files
```

## Build the binary

```sh
go build -o bin/orchestra ./cmd/orchestra
./bin/orchestra --help
```

Binary is at `bin/orchestra` (gitignored).

## Project structure

- `cmd/orchestra/` — Cobra CLI (main.go + one file per subcommand: run, do, plan, dashboard, agents, etc.)
- `internal/engine/` — the core supervised loop (dispatch → validate → retry → review). Every front door (`run`, shell, `do`) drives this.
- `internal/agent/` — Agent interface + registry. This is the extension point for adding new agents.
- `internal/router/` — AI routing: classifies intent → picks agent. Uses a CLI agent now; API planned.
- `internal/planner/` — decomposes requests into ordered steps with `depends_on` for parallelism.
- `internal/scheduler/` — bounded-concurrency DAG runner for parallel worktrees.
- `internal/worktree/` — git worktree isolation (branch per task, merge + conflict detection).
- `internal/validate/` — build → lint → test pipeline, auto-detects toolchain, stops on first failure.
- `internal/config/` — YAML config + built-in agent defaults + toolchain auto-detection.
- `internal/memory/` — SQLite history at `~/.orchestra/orchestra.db` (never in the repo tree).
- `internal/tui/` — Bubble Tea dashboard (agents, history, benchmarks, chat tabs).
- `internal/ui/` — terminal styling (gradients, spinners, diffs). TTY-aware.
- `internal/shell/` — interactive chat REPL.
- `internal/review/` — diff display + accept/reject prompt.
- `internal/runner/` — generic process exec (stream/capture/timeout). The engine's engine.
- `internal/gitutil/` — git helpers (is-repo, is-clean, diff, restore, commit).

## Key conventions

- **CGO_ENABLED=0** — pure-Go builds (uses `modernc.org/sqlite`, not CGo). Do not introduce C dependencies.
- **Tests use fake agents** — small shell scripts, not real CLIs. Tests are headless, offline, deterministic. Follow this pattern.
- **Test files are colocated** (`*_test.go` next to source). Pure logic (scheduler, config detection, TUI update/view) is tested without a TTY.
- **CI checks**: build, vet, test, gofmt. All four must pass. Run them locally before pushing.
- **Releases**: push a `v*` tag → GoReleaser cross-compiles → GitHub Release. Version stamped via `-ldflags -X main.version=<tag>`.
- **Config file**: `orchestra.yaml` (optional, generated by `orchestra init`). Built-in defaults work without one.
- **`dir_flag` in agent config**: some CLIs (e.g. opencode) don't honor process cwd. Set `dir_flag: "--dir"` so parallel worktrees stay isolated.

## Supervised loop (the core abstraction)

Every front door (`run`, `do`, shell) drives the same pipeline in `internal/engine/`:

```
dispatch → validate → retry (self-correct) → review (accept/reject)
```

If you change validation or retry logic, you change all three entry points. The engine is the product — not the CLI or the TUI.

## Agent interface contract

`internal/agent/agent.go` defines the extension point. Implement the `Agent` interface:

```go
type Agent interface {
    Name() string
    Run(ctx context.Context, task Task) (Result, error)
    Health() error
    Capabilities() []Capability
}
```

Optional interfaces that CLIAgent also implements: `Querier` (text output for planning), `Prober` (live health check), `QuietRunner` / `QuietQuerier` (no-stream variants for the TUI). Callers type-assert and fall back gracefully.

Register new agents in `internal/agent/registry.go` — the `Registry.Add()` method preserves insertion order for stable listings.

The only concrete implementation is `CLIAgent` (spawns a CLI in headless auto-approve mode). A future non-CLI agent (API-backed) can implement the interface without touching the scheduler, router, or shell.

## Testing with fake agents

Tests never call real agent CLIs. Instead, use small shell scripts or builtins as stubs:

```go
// "true" is a valid fake agent — exits 0, does nothing.
reg.Add(agent.New("alpha", "true", nil, "", []agent.Capability{agent.CapImplement}))
```

For scheduler tests, pure logic (DAG validation, concurrency caps) is tested without any agent at all. See `internal/scheduler/scheduler_test.go` for examples.

For config detection tests, temp dirs with marker files (`go.mod`, `package.json`) are used — no real toolchain needed beyond what's on PATH (tests skip gracefully via `t.Skip`).

## TUI stack

The dashboard (`internal/tui/`) uses Charm's ecosystem:
- **Bubble Tea** — Elm-architecture TUI framework
- **Lipgloss** — terminal styling
- **Glamour** — markdown rendering
- **Chroma** — syntax-highlighted diffs

These are not interchangeable with other TUI frameworks. Tests are headless (`tui_test.go` renders views without a TTY by injecting `WindowSizeMsg`).

## Router is CLI-based

The AI router (`internal/router/`) classifies intent by dispatching to a CLI agent (e.g. `claude -p`), not an HTTP API. It inherits agent CLI quirks: needs `--dangerously-skip-permissions`, uses the same `runner` package, and the classification latency is whatever the agent CLI takes. An API-based router is planned but not yet implemented.

## Gitignored directories

`code/` and `code2/` are local dogfooding test projects. They are gitignored — do not create or modify files there as part of your changes.
